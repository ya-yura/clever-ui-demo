# Ключевые артефакты: `spec.md`, `prompt_plan.md`, `todo.md`

Документ фиксирует состав и правила ведения трёх основных артефактов, которые требуются при работе над функциональностью в паре «инженер ↔ ИИ». Даже если файлы ещё не созданы, ориентируйтесь на следующие требования.

## 1. `spec.md` — функциональная спецификация
- **Назначение:** описывает, что именно должно быть реализовано (контекст, цели, сценарии, UX, API, ограничения).
- **Рекомендуемая структура:**
  1. _Overview_ — краткое описание задачи и бизнес-ценности.
  2. _User Stories & Flows_ — сценарии пользователя с шагами/точками интеграции.
  3. _UI/UX Notes_ — состояние экранов, макеты, требования к темам/скинам.
  4. _API & Data_ — источники данных, примерные payload’ы, ссылки на `$metadata.xml`.
  5. _Acceptance Criteria_ — чек-лист проверки готовности.
  6. _Non-Functional_ — производительность, оффлайн-режим, синхронизация, PWA‑ограничения.
- **Практика:** обновляйте файл при любом изменении требований, чтобы разработка и тестирование ссылались на единую правду. Если спецификация делится на подпроекты, разрешается использовать подразделы или ссылаться на другие документы из каталога `DOCS/`.

## 2. `prompt_plan.md` — план для работы с ИИ
- **Назначение:** пошаговое руководство, как разбирать задачу через ИИ‑агентов (Cursor, Claude, ChatGPT).
- **Типовые разделы:**
  1. _Goal Statement_ — какой результат нужно получить от ИИ.
  2. _Context to Provide_ — ключевые файлы/фрагменты кода, которые обязательно приложить к запросу.
  3. _Step-by-Step Prompts_ — последовательность подсказок (например: «1) распиши архитектуру…, 2) сгенерируй тесты…»).
  4. _Validation_ — что проверить после ответа (команды сборки, тесты, визуальный осмотр).
  5. _Fallback Strategy_ — что делать, если ответ неполный: уточняющие вопросы, дополнительные логи, метрики.
- **Практика:** файл помогает новым участникам команды повторять удачный сценарий общения с ИИ и не терять время на экспериментальные промпты.

## 3. `todo.md` — рабочий чек-лист
- **Назначение:** фиксация детализированных задач в рамках спринта/фичи.
- **Формат:**
  - Используйте маркированные списки с префиксами `[ ]`, `[x]`, `[~]` для статуса.
  - Каждая строка должна включать ссылку на файл/модуль или краткое описание ожидаемого результата.
  - Для зависимостей допускаются вложенные подпункты либо явные пометки «Блокирует: …».
- **Практика:** обновляйте файл при каждом существенном изменении плана (после ревью, новых требований, найденных багов). Это основной источник правды для асинхронной работы.

## 4. Где хранить и как поддерживать
- Все три файла располагаются в корне репозитория рядом с `README.md`. Если требуется разделить документацию по модулям, создавайте подпапки в `DOCS/` и добавляйте туда ссылки.
- При появлении новых версий файлов фиксируйте изменения в Git (не полагайтесь на устные договорённости).
- В README можно держать быстрые ссылки на актуальные артефакты, но полное содержание храните именно в `spec.md`, `prompt_plan.md`, `todo.md`.

## 5. Рабочий процесс
1. **Старт задачи:** заполните/обновите `spec.md`, чтобы зафиксировать требования.
2. **Подготовка промптов:** сформируйте последовательность действий в `prompt_plan.md`.
3. **Реализация:** ведите текущие шаги в `todo.md`; синхронизируйте статусы после каждого коммита или мерджа.
4. **Перед релизом:** убедитесь, что спецификация соответствует фактически реализованному поведению, план промптов сохраняет рабочий опыт, а todo либо закрыт, либо содержит явные переносы.

## 6. Что делать сейчас
Файлы ещё не созданы — заведите их при следующем большом изменении (например, при добавлении нового модуля или переработке логина) и следуйте правилам выше. Эта заметка служит отправной точкой: при необходимости дополняйте её ссылками на реальные артефакты и фиксируйте эволюцию процесса.
